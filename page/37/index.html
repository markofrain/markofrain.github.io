Vue2.0

项目结构与模块

Vuex并不限制代码结构，但如果你的store文件太大，getters和mutations这些对象里的东西太多，就搞得特别复杂，就像是多个业务模块冗余在一起。于是就有了模块。而它规定了一些规则。

  1. 应用层级的状态应该集中到单个 store 对象中。也就是说像是全局的数据之类的可以放在一个根store里，最后所有的module模块放在store里导出。

  2. 提交mutation是更改状态的唯一方法，并且这个过程是同步的。就是说你的数据更改必须要提交到mutation里修改，这样能保持过程同步，且更容易监视这个流程。

  3. 异步逻辑都应该封装到 action 里面。就如ajax或其他事件方法等需要放在action里。

那么官方的结构

├── index.html
├── main.js
├── api
│   └── ... # 抽取出API请求
├── components
│   ├── App.vue
│   └── ...
└── store
    ├── index.js          # 我们组装模块并导出 store 的地方
    ├── actions.js        # 根级别的 action
    ├── mutations.js      # 根级别的 mutation
    └── modules
        ├── cart.js       # 购物车模块
        └── products.js   # 产品模块

store文件夹下，action.js,mutations.js,actions.js就是用来放共用的或应用级的东西，最后在index.js中组装，而modules中都是些模块，cart.js和products.js里，都有action,mutations,actions，他们组成了单个模块。那么么他们也需要在index.js中组装并导出，所以就提供了一个modules的选项。

const moduleA = {
  state: { ... },
  mutations: { ... },
  actions: { ... },
  getters: { ... }
}

const store = new Vuex.Store({
  modules: {
    a: moduleA,
    b: moduleB
  }
})

modules选项中a,b代表模块名,moduleA和moduleB代表模块引用。 ** 导出js代码**

//模块里的导出
export default {
  state,
  getters,
  actions,
  mutations
}


//index.js
import Vue from 'vue'
import Vuex from 'vuex'
import action from './actions'
import getters from './getters'
import mutations from './mutations'
import user from './modules/user'

Vue.use(Vuex)

export default new Vuex.Store({
  action,
  getters,
  mutations,
  modules: {
    user
  }
})


//main.js,一定不要忘记设置store到Vue实例里
import store from './store/index'

你可以参考官网指出来的例子:购物车示例

模块的局部状态

  1. 对于模块内部的 mutation 和 getter，接收的第一个参数是模块的局部状态对象。也就是说在模块里js文件中,你获取的状态，参数都是局部的，也就是当前模块里的参数数据。

  2. 而对于模块里的actions,获取根节点状态则通过context.rootState

  3. 对于getters,根节点状态会作为第三个参数暴露出来

getters: {
    sumWithRootCount (state, getters, rootState) {
      return state.count + rootState.count
    }
  }

表单处理，双向绑定的计算属性

<input v-model="message">


// ...
computed: {
  message: {
    get () {
      return this.$store.state.obj.message
    },
    set (value) {
      this.$store.commit('updateMessage', value)
    }
  }
}

Vue2.0-Vuex插件学习

2018-09-21
Vue2.0

Vuex

Vuex 是一个专为 Vue.js 应用程序开发的状态管理模式。它采用集中式存储管理应用的所有组件的状态，并以相应的规则保证状态以一种可预测的方式发生变化。 按其他vue友所说,就是集中管理数据的，例如父子组件数据之间传递，如果项目很大复杂，那么这个就是一个问题，就需要用到它。可以说是一个数据共享的一个中心仓库。

基本使用

每个vuex的核心是store，意思上是一个容器，包含应用中大部分的状态。Vuex和单纯的全局对象的不同点:

  • Vuex的状态存储是响应式的。
  • 不能直接改变store中的状态

安装vuex

cnpm install vuex -S

store

既然是集中式管理，那么数据就存储在store中。 该图片解释了vuex的核心过程。

  • 首先通过组件调度发出(Dispatch)一个动作(Actions)
  • 随后动作提交到变化对象(Mutations)
  • 然后变化对象改变数据，数据都在State对象中
  • 数据改变后，重新渲染到组件中。

Vuex的核心是Store(仓库)，相当于一个容器，一个store容器中包含如下属性的方法

核心概念

释义

State

定义属性(状态、数据)

Getter

获取属性数据

Action

定义动作方法

commit

提交变化

Mutation

真正的变化

一、设置属性数据(关键state,getters)

1. 创建store.js并将store设置到main.js的Vue实例的store属性中

import store from './store.js'

new Vue({
  store
});

2. 编辑App.vue

定义两个button和一个count

<div id="app">
  <p>{ {count}}</p>
  <button>增加</button>
  <button>减小</button>
</div>

3. 编辑store，设置state,将state设置到store实例，并导出

// 引入组件
import Vue from 'vue'
import Vuex from 'vuex'
// 将vuex设置进来
Vue.use(Vuex)

// 设置state对象
const state = {
  count: 6
}
// 创建state对象
const store = new Vuex.Store({
  state // 设置state属性
})
// 导出store
export default store

4. App.vue中在computed选项中设置count，并通过this.$store.state.count返回数据

computed: {
  count () {
    return this.$store.state.count
  }
}

然后count值就出来了。 简洁的来说就是:

  1. 创建store.js文件，并引入到main.js文件中，作为store选项的值

  2. 在App.vue组件中设置要展示的数据

  3. 在store中引入vue和vuex,创建state对象，在其中设置共享数据。并创建store对象并设置state，然后导出

  4. App.vue中设置计算属性，通过this.$store.state.count获得值

** 我们还可以通过mapGetters,mapActions辅助函数,mapGetters用来获取属性数据，mapActions用来获取方法动作的。**

1. 要想使用，需要导入这两个函数,定义在App.vue的script标签中

import {mapGetters} from 'vuex'

2. 然后设置computed选项,设置为mapGetters函数，参数为数组，值为要获取的属性数据

computed: mapGetters([
  'count'
])

3. 那么这个count的值从哪里来，这就通过store的另一个概念getter。store文件中设置getters

// 设置getters对象
var getters = {
  count (state) {
    return state.count
  }
}

// 然后添加到store选项中

计算属性中的count值就与getters中的count名称对应，与state.count的count无关，他只是数据属性而已。

二、事件方法(关键action)

组件调度action去提交更改,action是定义方法的。所以需要action

1. 为App.vue中的button定义单击事件

<button @click="increment">增加</button>
<button @click="decrement">减小</button>

2. 在store中设置actions,并添加到store中

const actions = {
  increment (context) {
    console.log(context)
  },
  decrement (context) {
    console.log(context)
  }
}

3. 在App.vue中定义methods,需要通mapGetters一样导入mapActions

import {mapGetters, mapActions} from 'vuex'


methods: mapActions([
    'increment',
    'decrement'
  ])

actions中事件方法的参数context包含多个函数或对象，包括commit函数，dispath函数,getters对象,state对象等。 commit函数就用来提交改数据，actions指定定义事件方法，而更改在Mutations。

三、更改数据(关键，mutations)

1. 提交名称变化

const actions = {
  increment (context) {
    context.commit('increment') // 提交标识
  },
  decrement (context) {
    context.commit('decrement')
  }
}

2. mutations

const mutations = {
  increment (state) {
    state.count++
  },
  decrement (state) {
    state.count--
  }
}

mutations中的方法的参数就是state对象，可以通过state操作属性数据。 在这里，我们需要注意的是，mutations中的increment与actions中的commit函数的参数increment相同，他俩的值要相同不能更改.而actions中的increment方法名要和App.Vue中的methods中的mapActions里的值相同。 其实在actions中也是可以改state中的数据的，因为他的方法参数有state对象，但是并不建议这么做，而是把它放在mutations中做，它是专门做这件事的，而且这样更有利于devtools的查看，也符合vuex的设计思想。而actions中state要做的是什么?可以是对值的一个判断，是否去提交这个更改。 而getters，前面说了Getter是获取属性数据，所以i你可以定义方法来返回一个值，参数依然是state，然后你把方法名放在mapGetters中，就可以直接在模板中显示。 还有state，它通getters和actions一样，也有个mapState,可以放在Vue实例中。

3. 提交载荷和对象风格的提交方式

在和就是commit的参数，你在commit时，不仅可以传递一个type,还可以有一个附加参数。这个载荷会在devtools中显示出来.但大多数情况下载荷都是对象。

store.commit('increment', 10)

mutations: {
  increment (state, n) {
    state.count += n
  }
}

对象风格的提交就是，你可以直接在commit()中放置一个对象，而且是一个包含type属性的对象

store.commit({
  type: 'increment',
  amount: 10
})

mutations: {
  increment (state, payload) {
    state.count += payload.amount
  }
}

4. 使用常量代替Mutations事件类型

意思就是mutations中increment方法名需要改成一个常量，这个常量从专门定义的一个文件中获取

//type.js
const INCRENTMT = 'increment'
const DECREMENT = 'decrement'

export default{
  INCRENTMT,
  DECREMENT
}


//store.js
import types from '../type'
const mutations = {
  [types.INCREMENT] (state) {
    //...
  }
}
//或者
const actions = {
  increment ({commit,state}) {
    commit(types.INCREMENT)
  }
}

5. mutation必须是同步的

一条重要的原则就是要记住 mutation 必须是同步函数。 在devtools观察mutations时，会记录每一条mutations，能捕捉到前一状态后后一状态。而假如使用异步时:当 mutation 触发的时候，回调函数还没有被调用，devtools 不知道什么时候回调函数实际上被调用——实质上任何在回调函数中进行的状态的改变都是不可追踪的。

自定义全局组件

组件可分为两种:

  • 全局组件:使用Vue.use()全局引入，如:vue-router
  • 普通组件:需要每次引用组件,如:axios

插件通常会为 Vue 添加全局功能。插件的范围没有限制 Vue.js 的插件应该有一个公开方法 install。这个方法的第一个参数是 Vue 构造器，第二个参数是一个可选的选项对象。

export default{
  data(){
    return {
      msg:'one'
    }
  },
  install: function (Vue) {
    Vue.component('one', this)
  }
}

上面的代码是带当前vue文件中编写的install方法，所以Vue.component中使用了this关键字 可以单独定义index.js中写

import one from './one.vue'

export default{
  install:function(Vue){
    Vue.component('one',one);
  }
}

引入后直接通过名称one 你可以在其中定义全局方法或属性、全局资源，实例方法等,就官网所说

MyPlugin.install = function (Vue, options) {
  // 1. 添加全局方法或属性
  Vue.myGlobalMethod = function () {
    // 逻辑...
  }

  // 2. 添加全局资源
  Vue.directive('my-directive', {
    bind (el, binding, vnode, oldVnode) {
      // 逻辑...
    }
    ...
  })

  // 3. 注入组件
  Vue.mixin({
    created: function () {
      // 逻辑...
    }
    ...
  })

  // 4. 添加实例方法
  Vue.prototype.$myMethod = function (methodOptions) {
    // 逻辑...
  }
}

闪回技术:闪回表,闪回删表,闪回查询

2018-09-20
Oracle

闪回技术

闪回表(Flashback Table)

闪回表是利用UNDO表空间的撤销数据，所以能把表闪回到多久之前 受到undo_retention，UNDO表空间的数据文件是否启动自动增长功能，是否设置guarantee等三种因素的影响。 被闪回的表必须启动行移动功能，且该表必须存在

alter table dept enable row movement;

1. 闪回到具体时间

flashback table scott.emp to timestamp to_timestamp('2018-09-10 09:00:00','yyyy-mm-dd hh24:mi:ss')

  1. 闪回到10分钟之前

    flashback table scott.emp to timestamp(systimestamp-interval ‘10’ minute)

  1. 将scott.emp闪回到SCN为10086000的时候

    flashback table scott.emp to scn 1086000;

    select dbms_flashback.get_system_change_number scn from dual;—查看当前SCN值

使用闪回表注意事项:

  • 被闪回的表必须启动行移动功能,alter table dept enable row movement;
  • “FLASHBACK TABLE”命令的执行者必须有“FLASHBACK ANY TABLE”系统权限或者在被闪回的表上具有“FLASHBACK”对象权限。
  • “FLASHBACK TABLE”属于DDL命令，隐式提交。
  • SYS用户的任何表无法使用此功能。

闪回删表(Flashback Drop)

闪回删表指的是撤销“DROP TABLE”的效果。 1. 闪回被删掉的表scott.emp

flashback table scott.emp to before drop;

当表被删掉后，由新建了一个同名表，如果视图使用上面的闪回原表，则会报ORA-38312: original name is used by an existing object错误，可重新命名。 2. 闪回是可指被恢复的回收站对象

flashback table "BIN$AyId7ZbBjWngUKjADQIIuA==$0" to before drop;

闪回删表的工作原理是:当“DROP TABLE”命令执行时，表和索引并没有真正意义的删除，其所占空间只是分配给了回收站对象 注意：表空间在自动增长的压力下会按照先入先出的规则将回收站对象的空间分配给需要空间的段，在将回收站对象耗尽之前数据文件是不会自动增长的。 3. 禁用回收站功能

alter system set recyclebin='OFF' scope=spfile;

4. 删除当前用户回收站的所有对象

pruge recyclebin;

闪回查询

以表为单位查询过去的数据称为闪回查询，主要有两种方式：

  • 闪回时间点查询:利用select 命令的“as of”子句PL/SQL包dbms_flashback在过去的时间点上的查询。
  • 闪回版本查询:利用select命令的“version between”子句在过去的一段时间范围内的查询。

闪回时间点查询

利用as of语句

  1. 查询7900号员工在具体时间的工资

    select sal from emp as of timestamp to_timestamp(‘2018-09-13 10:02:30’,’yyyy-mm-dd,hh24:mi:ss’) where empno=7900;

  1. 查询7900号员工在5分钟之前的工资

    select sal from emp as of timestamp(systimestamp-interval ‘5’ minute) where empno=7900;

利用dbms_flashback包 利用dbms_flashback包的enable_at_time或enable_at_scn存储过程锁定一个会话级别的闪回时间目标，即进入闪回模式，随后的查询命令可以省略“as of”，直到调用dbms_flashback_disable存储过程将其关闭为止。 比如：将闪回模式回话定格在15分钟之前:

exec dbms_flashback.enable_at_time(systimestamp - interval '15' minute);

现在进行查询，注意，此时查询的是15分钟之前的表。

select sal from emp where empno=7900;   --忽略了“as of”子句

此时若访问SYSDATE、SYSTIMESTAMP等日期函数，它们的返回值仍是当前值，而不是15分钟之前的值。 处于闪回会话模式时，执行dml和ddl将报错

SQL> update emp set sal=2000 where empno=7900;
update emp set sal=2000 where empno=7900
       *
第 1 行出现错误:
ORA-08182: 在闪回模式下操作不受支持

如果查询完毕，可调用disable存储过程关闭闪回会话模式。

exec dbms_flashback.disable;

闪回版本查询 闪回版本查询可以贯穿一定长度的时间窗口，通过只使用一条查询命令就能返回该时间窗口内不同时间点上的数据。 比如，首先通过3个事务将7900号员工的工资进行修改。

SQL> update emp set sal=3000 where empno=7900;
已更新 1 行。
SQL> commit;
提交完成。
SQL> update emp set sal=4000 where empno=7900;
已更新 1 行。
SQL> commit;
提交完成。
SQL> update emp set sal=5000 where empno=7900;
已更新 1 行。
SQL> commit;
提交完成。
SQL> update emp set sal=6000 where empno=7900;
已更新 1 行。
SQL> commit;
提交完成。

执行闪回版本查询

SQL> select empno,sal from emp versions between timestamp(systimestamp -interval '15' minute) and maxvalue where empno=7900;

     EMPNO        SAL
---------- ----------
      7900       6000
      7900       5000
      7900       4000
      7900       3000
      7900        950

通过“versions between”，我们可以看到在15分钟之内，7900号员工的工资用5个值，说明共有4个事务对其进行过修改。 为了能看清这些事务的先后顺序，可以在查询列表中使用伪字段。

SQL> select versions_xid,versions_startscn,versions_endscn,empno,sal from emp versions between timestamp(systimestamp - interval '15' minute) and maxvalue where empno=7900 order by 2 nulls first;

VERSIONS_XID     VERSIONS_STARTSCN VERSIONS_ENDSCN      EMPNO        SAL
---------------- ----------------- --------------- ---------- ----------
                                           5482877       7900        950
0E00180005020000           5482877         5482880       7900       3000
17000D00E7010000           5482880         5482882       7900       4000
0F00140002020000           5482882         5482885       7900       5000
16000700C2020000           5482885                       7900       6000
  • versions_xid为事物号
  • versions_startscn是事物开始时的SCN
  • version_endscn时修改该行的下一个事物开始时的SCN

首尾衔接这两个字段的SCN号很容易得出真实的修改顺序。
