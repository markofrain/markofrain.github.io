问题描述

数据库、模式、角色、search_path、currentSchema

数据库

一个数据库可以对应一个应用，各个数据库是独立的。

角色与用户

角色是相对于整个数据库实例的，每个数据库都可以通过pg_roles查询到。几乎与用户相同，角色可以是一个用户create user命令等同于create role

除了一点:create user有隐含的login权限.而create role不会有该权限。也就是说通过create user创建，是可以作为用户名密码进行登录的。

创建用户

权限

权限有select,delete,update,insert,execute，可以针对各个表进行设置。

模式(命名空间namespace)

一个数据库下可以有多个模式。可以在创建模式的时候指定模式的所有者，或创建后alter更新所有者(用户角色)。

模式下可以创建任何对象，表、视图、索引、序列等。

每个模式下的表都是独立的，可以同名。

在同一数据库下，操作其他模式的表可以通过模式点表名查询。

模式与角色

模式可以被指定所有者角色owner。

一个角色可以绑定多个模式（当出现此情况，有同名表就需要额外注意）。

search_path

此参数作用于当前数据库。该参数表示当前查询的路径，值为模式名，默认是$user,pubilc，也就是说有了search_path里的模式，就不需要模式点表名了。

此时current_schema函数获取当前schema模式的值为search_path值的第一个模式名，需要注意的是search_path是有顺序的。

如果存在多个模式都search_path中，且多模式下有相同的表，如果不指定模式点的话，会从左到右模式一次查找表，将会匹配第一个。

因此这种操作必须使用模式点表名，否则如果你想要取的不在第一个就无法得到想要的数据，而且你需要时刻注意current_schema函数返回的值永远是第一个。

案例问题：

生产环境当有一个数据库，只有一个角色，多个模式的时候。我们在开发的时候是不同的角色的，多个数据库，一个角色给予一个模式。所以开发在正常的情况下切换多数据源。但是现场只有一个数据库，把原有的两个库的模式融合到一个库里了，且用户只有一个。

  1. 原有多数据源切换已经并没有任何作用，同一个库、同一个用户。切换相当于没切。但是因为开发环境下正常切换数据源无需使用模式点表名。但现场因为相当于没切数据源，那么就会查询public下的表，导致表不存在。

  2. 基于 生产环境无法改变一个库一个用户的情况。我们在不改动代码的情况下，将数据库的search_path由默认的public,增加到public,temp,current,history。此时我们就可以直接访问其他模式下的表了。

    但因为有些业务代码会查询current_schema函数获取切换数据源后的当前模式名，查询此模式名下所有符合条件的表名。这种情况直接拿到了public，导致没有拿到任何想要的匹配到的表。

  3. 为了解决这个问题。。。这个问题好像解决不了。

  4. 还有一个问题是，在开发环境下连到temp模式后，获取一个表的所有字段信息，我们只需要过滤角色即可，但现在所有模式都是一个角色，直接就查询除了所有模式下的同名表和字段。因此要想指定某个模式的表必须添加上过滤模式的条件。

  5. 另外对于search_path的顺序是至关重要的。

使用ALTER修改时，部分数据库，如翰高highgo，需要在未指定模式下执行才生效。

给用户角色增加search_path属性

添加完成后的配置在pg_roles中的rolconfig字段中会显示出来。然后java应用进行数据库连接后，所指定用户名密码登录的用户角色就使用这个配置。

这突然让我不知道，ALTER DATABASE db set seatch_path to pubilc是有什么作用了。是否应用于整个数据库？

当一个数据库只有一个用户角色的情况下，这种alter user设置的和alter database设置的效果应该是一样的。

测试了一下

使用ALTER DATABASE sxkfk set search_path to public,business,temp,current;似乎没有任何效果，alter user是有效果的。

但是使用alter user sxkf reset search_path来重置search_path，虽然pg_role看不到了。但是已经设置过的还在，没有重置，使用alter database sxkfk reset search_path确还原了，还原成默认"$user",public。

以上测试问题，通过postgresql.conf的参数设置章节了解到，配置文件<alter database<alter user<set这种优先级。

重加载配置

pg_ctl reload命令会重新读取这个配置文件，同时主服务器进程也将这个信号广播给所有正在运行的子服务器进程，这样现有会话也能得到新值。有些参数只能在服务器启动的时候设置；对这些条目的修改将被忽略，直到下次服务器重启。

对于单个数据库或角色的更新覆盖

  1. 我们可以给一个用户或者一个数据库赋予一套选项设置。在一个会话开始的时候，装载所涉及到的用户和数据库的缺省设置。命令 ALTER USER 和 ALTER DATABASE 分别用于配置这些设置。针对每个数据库的设置将覆盖任何从 postgres 命令行或者配置文件收到的设置，然后接着又被针对每个用户的设置覆盖；最后又会都被针对每个会话的设置覆盖。

    也可以把一个会话缺省绑定到一个特定用户，而不是某个数据库上；参阅 ALTER USER ，如果存在冲突，那么用户声明的设置覆盖数据库相关的设置。
