JVM 参数

参数 说明
-Xmx / -XX:MaxHeapSize 堆内存最大大小
-Xms / -XX:InitialHeapSize 堆内存初始大小
-Xmn 新生代大小
-XX:+HeapDumpOnOutOfMemoryError 当出现内存溢出时Dump出当前内存堆转储快照文件
-Xss 栈大小
-XX:MetaspaceSize= 元空间初始大小，以字节为单位
-XX:MaxMetaspaceSize= 元空间最大值，默认是-1，只受限于本地内存大小
-XX:MinMetaspaceFreeRatio= 在垃圾收集后控制最小的元空间 剩余容量的百分比
-XX:MaxMetaspaceFreeRatio= 控制最大的元空间 剩余容量的百分比
-XX:MaxDirectMemorySize=10M 最大直接内存大小
-Xloggc:gc.log 输出gc日志到文件，目录需提前存在
-XX:+PrintGC 控制台打印GC日志，配置了输出文件，则不会控制台打印
-XX:PretenureSizeThreshold=3145728 指定大于该值的对象直接在老年代分配。只对Serial和ParNew两款新生代收集器有效
-XX:+PrintGCDetails 打印GC收集器日志
-XX:SurvivorRatio=8 决定了新生代中Eden区与一 个Survivor区的空间比例是8∶1
-XX:MaxTenuringThreshold=1 对象晋升老年代的年龄阈值

Spring源码理解-Spring Cache CacheNames属性

2021-11-12
Spring

因为测试了SpringCache的使用，@Cacheable属性cacheNames的值设置了hello，配置类里也配置了CacheManager，因为是简单测试，就使用了基于ConcurrentMap的缓存存储方式。

对于缓存方式的实现，可以有看org.springframework.cache.Cache接口的实现类，都是缓存的实现类型。Spring提供的包括

NoOpCache：无缓存实现，仅仅是提供方法，不进行任何存储

EhCacheCache：基于EhCache的缓存实现

ConcurrentMapCache：基于ConcurrentMap类的缓存实现

JCacheCache：基于JCache的缓存实现

CaffeineCache：基于Caffeine的缓存实现

其他外置的包括RedisCache，RedissionCache等等

而缓存管理器CacheManger，由AbstractCacheManager抽象类实现，大多都是直接或间接继承AbstractCacheManager类或实现CacheManger接口。

此处在AppConfig配置类中，通过Bean方式注册了SimpleCacheManager，使用了ConcurrentMapCacheFactoryBean的方式。因为使用了@Cacheable的cacheNames属性，因此ConcurrentMapCacheFactoryBean中的name必须要有，他是一类缓存的名字。如果在缓存管理器中没有找到对应的缓存名称就会出现问题。而这仅是SimpleCacheManager才会出现的问题。也就是说它必须要手动指定name，如果有多个就创建多个ConcurrentMapCacheFactoryBean。

只因为它并没有实现AbstractCacheManager的getMissingCache方法，表示当没有获取到对应名称的缓存对象的时候该如何返回一个正确的Cache对象。

因为在使用RedisCache的时候，默认是实现了这个方法的，所以在使用的时候不需要手动定义缓存的名称，它是动态的，不存在就创建，然后存储起来。

对于缓存的获取是在抽象类AbstractCacheManager中，他有个getCache方法。获取Cache时会从map中拿，如果没有就调用getMissingCache()方法，然后获取到之后执行decorateCache()方法，进行其他需要做的事，此处是直接返回了。然后就put到集合中，再更新cacheNames集合。

这里面的cacheMap属性就存储了多个名称对应Cache对象的一个map集合。cacheNames是所有缓存名称的集合

而SimpleCacheManager就非常的简单，自定义一个caches，仅仅是实现了一个必须要实现的方法loadCaches，就是多个Cache对象。这个会交给父类统一管理到cacheMap中
